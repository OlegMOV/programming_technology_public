НАВЧАЛЬНІ ПИТАННЯ

1. Поняття та принципи роботи систем контролю версій

Теорія - 20 хв, практика - 0 хв

Система контролю версій (далі - СКВ) - це система, що записує зміни, які відбулися у файлі/ах протягом деякого часу, та забезпечує можливість повернення до конкретної версії файлу/файлів.

Файли не обов'язково мають бути з текстом програм. Це можуть бути будь-які файли.

СКВ дозволяє підвищити шанси не втратити важливу для вас інформацію, а також зафіксувати історію зміни даних у проекті, який відслідковується.
Крім того, СКВ дозволяє організувати спільну роботу учасників проекту. Для цього є відповідні механізми, що дозволяють поєднувати результати роботи багатьох виконавців.

1.1 Основні поняття
До основних термінів та понять слід віднести:

  • репозиторій (repository) - сховище де зберігаються дані СКВ;
  • гілка (branch) - відокремлена історія розвитку змін файлів у проекті;
  • знімок (commit) -зафіксовані та збережені у СКВ стани файлів проекту.

Репозиторій - це база даних, де зберігається історія змін у файлах - станів. А також інструменти, які дозволяють здійснювати маніпуляції з цими станами.

Гілка - відокремлена лінія розвитку (змін змісту файлів) файлів проекту. Що дозволяє, відокремившись в окрему гілку, експериментувати зі змінами файлів. Зазвичай назви гілок є такими:
- main або master - тут зберігається робоча версія коду програми
- bugfix - для виправлення помилок, виявлених у коді програми
- hotfix - термінові виправлення, які виявлені у версії програми, яка вже використовується замовником
- feature/ -

Знімок - зафіксований у репозиторії стан файлів проекту. Під станом розуміється зміст файлів.

За принципом дії СКВ розділяють на:

  • локальні - такі, що функціонують на одній ПЕОМ. Вони не підтримують командної роботи;
  • централізовані - мають сервер, де і зберігаються усі зміни. Втрата доступу до серверу - значить втрата усієї історії розвитку проекту;
  • розподілені - дані з історією змін знаходяться у кожного учасника проекту. Це підвищує надійність такої системи

Зазвичай принцип роботи СКВ є наступним:

Рис.1.1

Для кожного знімку проекту зберігаються тільки зміни, що були внесені на кожному кроці, і цим відрізняють версії.

Рис.1.2

Система `Git` зберігає у кожній версії проекту усі файли заново. З однією лише оптимізацією - незмінений файл у даній версії зберігається як посилання на попередній знімок (*позначено курсивом на рис. 1.2*).

У подальшому будемо розглядати роботу із Git.

1.2 Принцип роботи з Git.

  1. Кожен новий файл, який створено/переміщено у теку проекту, має стан untracked.
  2. Для того щоб СКВ почала роботу по відслідковуванню змін цього файлу необхідно його додати до списку (git add path_to_file). Після чого стан файлу змінюється на staged.
  3. При прийняттю рішення щодо готовності файлу/ів до збереження, наявний зміст файлу/ів зберігають у репозиторій (git commit). Після чого стан файлу змінюється на commited.

Для цього використовуються наступні команди

Рис.1.3

При внесенні файлів під контроль СКВ у подальшому вони вже відслідковуються. При внесенні у них змін їх стан змінюється на `modified`, а подальші команди залишаються такими ж (`modified`-> `staged`-> `committed`).

2. Встановлення та налаштування

Теорія - 10 хв, практика - 20 хв

Перед початком використання Git, його потрібно встановити на ПЕОМ.

Встановивши Git, у подальшому необхідно періодично його оновлювати

Для встановлення, необхідно (приклад для ОС Windows):

  1. зайти на офіційну сторінку http://git-scm.com/download/win;
  2. завантажити на ПЕОМ дистрибутив;
  3. встановити (виконання не потребує додаткових знань)
  4. налаштувати.

2.1 Налаштування

Встановивши Git, є необхідність у його початковому налаштуванню. Для цього використовується утиліта config.
Для цього у командному рядку (comand prompt,powershell, bash, etc) виконують наступні команди:

  1. git config --global user.name "nameUser" - на місці nameUser необхідно вказати ваш nick - який буде ідентифікувати вас як користувача СКВ.
  2. git config --global user.email "user@email.com" - на місці user@email.com ваш e-mail.
  3. git config --global color.ui true - підключення файлу з кольоровою схемою для командної строки.
для nameUser необхідно використовувати латинські літери
зазначена інформація у подальшому використовується для позначення авторства щодо внесення змін у репозиторій СКВ

Ввівши дані є можливість перевірити їх значення командою git congig --list

2.2 Створення репозиторію

Після налаштування, СКВ готова до використання. Для цього необхідно:

  1. створити теку у якій буде реалізовуватися процес відслідковування змін. Це доцільно зробити через командний рядок mkdir nameDir, де nameDir назва теки
  2. увійти у створену теку cd nameDir
  3. сповістити встановлену в ОС СКВ щодо того, що у заданій теці має реалізовуватися контроль за зміною версій файлів git init . За цією командою створюється прихована тека .git де розміщуються файли, що реалізують логіку роботи Git.

Практичне завдання 1

  1. У середовищі windows sandbox встановити СКВ.
  2. Здійснити початкові налаштування облікового запису.
  3. Створити теку та активувати Git.

2.3 Створення файлу виключень

Як правило тека де розробляється проект містить службові файли, та і такі, що не несуть цінності (тимчасові файли). Це також можуть бути і теки. Для цього є можливість створити файл із їх переліком. Це позбавить від необхідності кожного разу відслідковувати чи не попали вони у список tracked. Крім того, це дозволить застосовувати команду git add .
Для цього:

  • у теці проекту створити файл із назвою .gitignore (по суті своїй - текстовий файл)
  • відкрити його у будь-якому текстовому редакторі
  • записати перелік тек та файлів, які не мають відслідковуватися
кожен запис із нового рядка: для тек - name_dir/ , для файлів - name_file.docx

3. Базові операції

Теорія - 15 хв, практика - 20 хв

Для роботи із СКВ Git достатньо, у переважній більшості часу, використовувати нечисельний набір основних команд.

  1. Перевірка стану (статусу) файлів проекту:
    git status

У випадку коли змін не відбулося, має бути наступна інформація:

	PS D:\mov_vault> git status
	On branch master
	Your branch is ahead of 'origin/master' by 2 commits.
	  (use "git push" to publish your local commits)

	nothing to commit, working tree clean

Рис.3.1

В іншому випадку виводиться у командний рядок стан змінених/доданих файлів, наприклад:
PS D:\mov_vault> git status
Your branch is up to date with 'origin/master'.

Changes to be committed:
(use "git restore --staged <file>..." to unstage)
new file: ^q1.md
new file: developing/Learning/PT/31_GL/group_lesson_git.md

Changes not staged for commit:
(use "git add/rm <file>..." to update what will be committed)
(use "git restore <file>..." to discard changes in working directory)
deleted: ^q1.md
modified: developing/Learning/PT/31_GL/group_lesson_git.md

Рис.3.2

Що дозволяє побачити (останні рядки), - файл `^q1.md` - був видалений, - файл `group_lesson_git.md` - змінений.
  1. Підготовка файлів до фіксування змін реалізується командою:
    git add nameFile1 name File2, де nameFile1 name File2 - файли, які мають бути переведені у стан staged.
якщо усі файли мають увійти у список таких, що відстежуються СКВ, застосовується команда - git add ., де . - маска для усіх файлів
щоразу перед фіксуванням змін необхідно виконувати команду git add , відповідно з вказанням конкретних файлів або маски .
  1. Фіксування змін файлів у репозиторії реалізується командою:
    git commit. Це дозволяє створити знімок поточного стану файлів у активованому репозиторії.
    Для більш зручного використання репозиторію є можливість створення текстового заголовку до знімку:
    git commit -m "Title for commit" - де на місці "Title for commit" має бути текст, що описує внесені зміни.
  2. Перегляд історії збереження станів
    git log, що дозволяє побачити перелік зафіксованих станів
	PS D:\mov_vault> git log
	commit b5720dacdf6cbe3a3da880e892fc4afccd626fb1
	 (HEAD -> master, origin/master, origin/HEAD)
	Author: OlegMOV <surez@meta.ua>
	Date:   Wed Aug 27 12:58:55 2025 +0300

	    vault backup: 2025-08-27 12:58:54

	commit 002ec40fbbf40ad0fdbf47d9ced254be3dc0bc29
	Author: OlegMOV <surez@meta.ua>
	Date:   Wed Aug 27 11:15:11 2025 +0300

	    vault backup: 2025-08-27 11:15:11

	commit 13a3af31b944b827c7332089a86b08b83448f8b5
	Author: OlegMOV <surez@meta.ua>
	Date:   Wed Aug 27 11:14:55 2025 +0300

	    vault backup: 2025-08-27 11:14:55

Рис.3.3

Тут вказано зафіксовані стани, для кожного із них система генерує хеш-суму commit b5720dacdf6cbe3a3da880e892fc4afccd626fb1, дату коли зафіксовано стан, та де знаходиться показник поточного стану (HEAD -> master, origin/master, origin/HEAD).

для виходу із перегляду історії станів необхідно натиснути клавішу Q
  1. Для повернення до попередніх зафіксованих станів файлів, використовують команду:
    git checkout HEAD^^^ або git checkout HEAD~3 або git checkout <hash>

Розглянемо на рисунку

Рис.3.4

Зафіксовані останні зміни у точці `Coomit_3`, там знаходиться мітка СКВ `HEAD`. У випадку необхідності переглянути попередні редакції будь-якого файлу проекту, є можливість переміститись у той стан. Наприклад, якщо необхідно переміститися у стан, що зафіксований під назвою `Commit_1` - потрібно мітку `HEAD` змістити **на дві позиції**. А це можна реалізувати трьома способами: 1. `git checkout HEAD^^` де кількість `^` відповідає необхідній кількості позицій, наприклад на 3 позиції `^^^`; 2. `git checkout HEAD~2`, де число `~2` - відповідно 2 позиції 3. `git checkout `, де `` хеш зафіксованого стану.

Хеш зафіксованого стану <hash_commit> можливо подивитися викликом команди git log.
При переході в інші стани репозиторію зміщується на вказаний стан

Рис.3.5

Переглянувши файли, ми не можемо вносити зміни, оскільки СКВ знаходиться у режимі `detached`. Для повернення на останній стан необхідно ввести команду `git checkout main`, де `main` - назва гілки.

Практичне завдання 2

  1. Створити файл у теці проекту: main.py.
  2. У створеному файлі записати код програми, який створює файл data.txt та записує у нього текстову інформацію (Два речення).
  3. Додати файл під контроль версій та зафіксувати перший стан з описом змін.
  4. Внести зміни у main.py (наприклад, змініть текст для запису у файл) та зафіксувати другий стан.
  5. Переглянути історію станів за допомогою команди git log.
  6. Повернутися до попереднього стану проекту за допомогою команди git checkout <hash>.
  7. Повернутися на основну гілку.
